D1: User Support & Workflows
15%
▼
Admins handle config, access, integrations. GitHub Support handles billing, bugs, outages.
Generate support bundles via ghe-support-bundle command on GHES or through enterprise admin panel.
GitHub App vs Action: Apps are independent services with fine-grained permissions and webhooks. Actions are CI/CD automation steps that run in workflows.
Fork-and-pull = contributors fork the repo and submit PRs. Branching = all team members work in branches of the same repo.
Branch protection rules: require PR reviews, status checks, signed commits, restrict who can push.
GitHub APIs allow programmatic access to audit logs, org metrics, and admin tasks beyond the UI.
💡 Key: GitHub Marketplace provides Apps and Actions. Apps require installation on org/repo. Know the difference between OAuth Apps (user-auth) vs GitHub Apps (app identity with fine-grained permissions).
D2: Identities & Authentication
20%
▼
SAML SSO: Delegates authentication to your Identity Provider (IdP). Users must authenticate via IdP to access org resources. Can be enabled per-org or enforced at enterprise level.
SCIM: System for Cross-domain Identity Management. Automates user provisioning/deprovisioning between IdP and GitHub. Supported providers: Azure AD, Okta, and custom implementations.
Team Sync: Links GitHub teams to IdP groups. When group membership changes in IdP, GitHub team membership updates automatically. Works with Azure AD and Okta.
SCIM vs Team Sync: SCIM manages user accounts (provision/deprovision). Team Sync manages team membership specifically.
2FA: Org owners can require 2FA for all members. Users without 2FA are removed from org. Supported: TOTP apps, SMS (limited), security keys (WebAuthn).
Enforcing SAML at enterprise level requires all member organizations to use the same IdP.
💡 Key distinction: SAML SSO = Authentication (who are you?). SCIM = Provisioning (create/update/delete user accounts). Team Sync = Authorization grouping (which teams?).
D3: Deployment & Licensing
5%
▼
GHEC (GitHub Enterprise Cloud): SaaS, hosted by GitHub. Supports SAML SSO, audit log streaming, enterprise policies. Best for orgs wanting cloud convenience.
GHES (GitHub Enterprise Server): Self-hosted, runs on your infrastructure (or cloud VMs). You control updates, backups, and networking. Ideal for strict data residency requirements.
GHAE (GitHub AE): Fully managed by GitHub but isolated instance. Single-tenant. Deprecated/no longer sold for new customers.
GitHub Actions pricing: Free minutes per plan, then charged per minute. Linux, Windows, macOS runners have different multipliers (macOS costs ~10x Linux).
License usage: tracked per unique user across all orgs in the enterprise. Machine/bot accounts with push access consume licenses.
💡 GHES = you manage infrastructure. GHEC = GitHub manages infrastructure. Key admin difference: GHES requires manual upgrades and backups.
D4: Access & Permissions
20%
▼
Org Roles: Owner (full control), Member (access per settings), Billing Manager (billing only, no code access), Outside Collaborator (access to specific repos only, not org member).
Repo Roles: Read, Triage, Write, Maintain, Admin. Each grants progressively more capability.
Team permissions: Teams can be granted repo access. Child teams inherit parent team's repo permissions. Team maintainers can manage team membership.
Default member permissions can be set to None, Read, Write, or Admin at the org level. Best practice: set to Read or None, grant more via teams.
Single org vs multiple orgs: Single = simpler management, broader visibility. Multiple = stronger isolation, more admin overhead.
Outside collaborators don't count as org members for SAML enforcement and have limited org visibility.
💡 Least privilege principle: Grant minimum access needed. Use teams to manage permissions at scale. Repository admins can't exceed organization policies.
D5: Security & Compliance
15%
▼
Secret Scanning: Detects accidentally committed tokens/keys. Alerts repo admins. Partners with providers (AWS, GitHub, etc.) to immediately revoke found tokens.
Code Scanning (CodeQL): Static analysis that finds security vulnerabilities in code. Can be set up via GitHub Actions workflow. Results appear as alerts on PRs and in Security tab.
Dependabot: 3 features — Dependency Graph (visualize deps), Security Alerts (notify on vulnerable deps), Version Updates (auto-open PRs to update deps).
Audit Log: Records all significant actions in the org/enterprise (who did what, when). Accessible via UI, REST API, or GraphQL API. Can stream to external SIEM.
Token types: PAT (Personal Access Token) — user-level, broad. Fine-grained PAT — scoped. OAuth token — for OAuth Apps. GitHub App Installation Token — short-lived, least privilege. Device token, Refresh token.
EMU (Enterprise Managed Users): User accounts are created and managed entirely by the enterprise IdP. Users cannot create personal accounts or access outside the enterprise.
Remove sensitive data: git filter-branch or BFG Repo Cleaner for local repo. Contact GitHub Support to purge from their caches.
💡 GitHub Apps > PATs for automation. Apps have fine-grained permissions, short-lived tokens, and audit trail per-app. PATs are broad and user-tied.
D6: GitHub Actions
20%
▼
GitHub-hosted runners: Managed by GitHub, auto-scaled, ephemeral. Available in Ubuntu, Windows, macOS. No maintenance required but limited customization.
Self-hosted runners: You manage the machine. More customization, can access internal resources, no per-minute cost. Risk: public repos can trigger arbitrary workflow runs on your runner.
Runner groups: Organize runners and control which orgs/repos can use them. Enterprise-level groups can span orgs. Prevent unauthorized access to runners.
Encrypted secrets: Scopes — Repository, Environment, Organization, Enterprise. Org/Enterprise secrets can be set to "all repos" or "selected repos". Secrets not passed to forked repos by default.
Reusable workflows: Defined with workflow_call trigger. Allow standardizing CI/CD across org. Caller workflows reference with uses: org/repo/.github/workflows/file.yml.
IP allow lists: GHEC orgs can restrict incoming connections. GitHub-hosted runners have dynamic IPs — you must use the hosted runner IP ranges or disable IP allow lists for those workflows.
💡 Security risk: Never use self-hosted runners on public repos — a forked PR can modify workflow files and execute malicious code on your runner.
D7: GitHub Packages
5%
▼
Supported registries: npm, RubyGems, Apache Maven, Gradle, Docker (Container Registry), NuGet.
Packages are tied to a repository or organization. Permissions follow repository permissions by default.
Use GITHUB_TOKEN in Actions workflows to authenticate and publish packages without storing secrets.
Packages vs Releases: Packages = versioned software artifacts stored in a registry (npm, Docker). Releases = Git tags + release notes + attached binary assets. Releases are human-facing; Packages are dependency-facing.
💡 GitHub Packages storage counts toward plan limits. Actions minutes used to build/push packages also count.
